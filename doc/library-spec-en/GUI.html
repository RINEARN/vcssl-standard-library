GUI

Abstract

The VCSSL GUI library is the standard library for handling graphical user interfaces in VCSSL.

Because the GUI library is relatively large, it is difficult to use it only by referring to the function specifications described here.For concrete usage examples, see the Official VCSSL GUI Development Guide.

- Note -
In programs using a GUI, event handlers are essential. In VCSSL, event handlers are executed in independent threads.The current version of RINEARN VCSSL Runtime does not yet provide sufficient support for multithreaded processing. This is because the current basic design of the VCSSL engine still partially follows the design of early VCSSL engines, which were developed assuming only single-threaded execution.As a result, depending on the processing performed, several issues including logical inconsistencies may occur. When writing event-handler processing in RINEARN VCSSL Runtime, do so with this situation in mind.At present, the following kinds of issues may occur.
- Inconsistent indexes when the same array variable is accessed simultaneously from multiple threads
In the current VCSSL engine, some modules that access array-variable values still retain designs intended only for single-threaded execution. Because of this, if the same array variable is accessed simultaneously, incorrect index values may be referenced.
- Emergency shutdown of the engine caused by extremely frequent simultaneous multithreaded access to the same function
In principle, the same function can be executed by multiple threads. In that case, arguments and local variables are secured separately for each thread so that they are not mixed up between threads. However, if simultaneous access is concentrated within too short an interval, this processing may fail to keep up, and the engine may perform an emergency shutdown to prevent inconsistencies. Overlap in function execution time is not the issue. The problem is when the moments of calling the function, that is, passing arguments, overlap.
Because issues such as the above may occur, when writing processing inside event handlers in the current RINEARN VCSSL Runtime, design it with the above points in mind and follow guidelines such as the following.
- In event handlers, keep processing lightweight, such as storing event information in global variables, and perform the actual processing in the main thread whenever possible

- Do not access array variables that are frequently accessed in the main thread from event handlers or other threads
However, if the main thread, that is, the thread executing the automatically started processing at launch, finishes after only short tasks such as constructing the GUI screen, and thereafter almost all processing occurs only from GUI event handlers in response to user operations, then the above points do not need to be considered very much.
