Graphics3DFramework

Abstract

This library, graphics3d.Graphics3DFramework, is a lightweight framework for handling 3D computer graphics easily.

When writing a 3D CG program from scratch using only the standard VCSSL libraries Graphics3D and GUI,
you need to write several fixed pieces of code every time, such as creating the window and renderer, the animation loop,and basic event handlers.
Graphics3DFramework supports easy 3D CG development by preparing such fixed processing in advance as a framework template.

Required Processing

Structurally, Graphics3DFramework is already a complete 3D CG program by itself, and it can actually run as a standalone program.
However, by itself no 3D models are placed, so only a white screen is displayed.

In practice, you import Graphics3DFramework from your own module and describe the required processing, such as placing and moving 3D models.
Such processing is written by defining functions with specific names, just like event handlers for GUI components.
Then Graphics3DFramework automatically executes those event handler functions at appropriate timings.

Graphics3DFramework automatically executes the following event handler functions. Their definitions are optional, so define only the ones you need.

  • void onStart( int renderer ) - Executed only once at startup. Write initialization processing such as creating and placing 3D models here. The old name initialize is also supported.
  • void onExit( int renderer ) - Executed only once at shutdown. Write any shutdown processing here if needed. The old name finalize is also supported.
  • void onPaint( int renderer ) - Executed every time the screen is updated, usually dozens of times per second. Write processing here when you want to draw foreground or background 2D CG contents. The old name paint is also supported.
  • void onUpdate( int renderer ) - Executed every time between screen updates, usually dozens of times per second. Write processing here when you want to move 3D models and so on. The old name update is also supported.
  • void onResize( int renderer ) - Executed when the window size changes. Write processing here when you want to adjust the layout of GUI components placed on the screen. The old name resize is also supported.
In all of the above functions, the argument renderer receives the ID of the 3D CG renderer. Use it for placing 3D models and related processing.

Background and Foreground 2D CG Drawing

Graphics3DFramework also supports foreground and background 2D CG drawing.
However, this feature is disabled by default, so before using it, callsetForegroundRendererState ( true ) andsetBackgroundRendererState ( true ) respectively.
After that, obtain the IDs of the 2D CG renderers withgetForegroundRenderer ( ) andgetBackgroundRenderer ( ), and perform drawing in functions such as onPaint.

Window Customization

Graphics3DFramework allows margin areas to be set inside the window, where GUI components such as buttons and text fields can be placed.
Margins can be configured with functions such as setMarginTop ( int margin ) and setMarginLeft ( int margin ).
Then obtain the window ID by getWindow ( ) and place GUI components freely in the margin areas.

Frame Rate Adjustment

Graphics3DFramework also has an automatic frame rate adjustment mechanism.
Specifically, it measures the time required for each frame and automatically adjusts the wait time of the animation loop so that the measured frame rate approaches the configured target frame rate.
Set the target frame rate with setFrameRate ( float rate ).The measured frame rate can be obtained with getCurrentFrameRate ( ).

Slowdown

Depending on the processing speed of the execution environment, if the drawing area is large or the drawing contents are heavy,
the measured frame rate may fail to reach the target frame rate even when the animation wait time is minimized. This phenomenon is a slowdown.

When a slowdown occurs, screen repainting may fail to keep up and the display may flicker.
In that case, flicker can be reduced by setting the target frame rate to a value with enough margin below the measured frame rate limit.
However, because the measured frame rate depends heavily on the environment,flicker can also be reduced by setting a relatively large lower limit for the animation wait time in milliseconds with setAnimationWaitMin ( int wait ).
Note that this lowers the upper limit of the measured frame rate.
